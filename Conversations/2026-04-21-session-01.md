## Idea: PDF overlay export (not built — documented for future)

Discussed replacing the summary sheet export with filling answers directly onto the original PDF. Three approaches considered:

### Option A — AcroForm filling
Use `pdf-lib` to detect and fill real AcroForm fields by name. Problem: PDF field names are often opaque, and many government forms are scanned images with no AcroForms.

### Option B — Gemini coordinate overlay
Change the scan prompt to return bounding box coordinates for each field alongside labels. Use `pdf-lib` to draw text at those coordinates on the original PDF. Gemini supports bounding box return (0–1000 normalized scale). Requires coordinate mapping to PDF point space.

### Option C — Hybrid (preferred approach if we build it)
At upload time, use `pdf-lib` to extract AcroForm field names (if present) and pass them into the Gemini scan prompt: *"The PDF has these fields: [list] — use these as your `id` values."* This gives a guaranteed 1:1 mapping at export with no fuzzy matching needed. Falls back to summary sheet if no AcroForms detected.

**Difficulty:** Medium. Main risks: PDF field names in the wild vary from clean (`First_Name`) to opaque (`topmostSubform[0].Page1[0].f1_01[0]`). XFA-based PDFs (some federal forms) not supported by `pdf-lib`.

**Decision:** Defer. Test domain history and current export in production first. Worth revisiting once core flow is stable.
