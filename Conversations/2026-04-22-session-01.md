### Gemini 503 reliability — root cause found and addressed

Consistent 503 errors traced through several failed attempts before finding root cause:

**What we ruled out:**
- API key validity — confirmed working via direct curl (simple text requests succeed)
- Payload size — file URI approach (Gemini Files API) still 503d on generate call
- Model deprecation — `gemini-2.0-flash` is 404 ("no longer available to new users"), `gemini-1.5-flash` also 404 for this key

**Root cause confirmed:**
Google's own error message: *"This model is currently experiencing high demand. Spikes in demand are usually temporary."* — `gemini-2.5-flash` multimodal calls were genuinely overloaded.

**What we built to address it:**

1. **Gemini Files API upload** (`netlify/functions/gemini.js` + `server.js`): Instead of sending the full base64 PDF inline, we upload the file to Gemini's Files API first (`?action=upload`), get a URI, and send only the URI in the generate call. Reduces generate request payload from ~5MB to ~1KB.

2. **AbortSignal timeout (22s)** in the Netlify function: Prevents the CDN "Inactivity Timeout" HTML page that appeared when Gemini was slow — now returns a clean 503 JSON response before the CDN cuts the connection.

3. **Smarter retry UX in `initPhase2`**: Instead of silent failure, the scan phase now auto-retries with a visible countdown (8s → 20s → 40s). User sees `"AI service is busy — retrying in Xs…"` rather than a spinner that gives up.

4. **Save-on-upload**: After the file uploads to Gemini's Files API, `saveFormState('scan')` is called immediately. A green "Progress saved — you can close this and come back later." note appears below the scan brief. If the user closes and returns, the restore flow skips the upload step and goes straight to the scan attempt using the saved `fileUri`.

**New stage:** `saved_forms.stage` can now be `'scan'` (in addition to `'fill'` and `'review'`). `restoreFormState` handles this by entering phase 2 with `state.fileUri` already set.

### AcroForm extraction (PDF.js) — skip Gemini for fillable PDFs

Implemented a hybrid scan path:

1. If the uploaded file is a PDF, attempt AcroForm field extraction using **PDF.js** (loaded lazily from CDN).
2. If fields are found, skip the full Gemini multimodal scan entirely.
3. Make a small **text-only** call to **`gemini-2.5-flash-lite`** with just the field IDs to get domain, docType, summary, and profile-based prefill suggestions.
4. If the text call fails, fall back gracefully to `domain='Other'` and the filename as docType — fields are always available regardless.
5. If no AcroForms found (scanned/flat PDF or image), fall through to the existing Gemini multimodal path.

**Why PDF.js over pdf-lib:** pdf-lib threw on "Invalid object ref" warnings from malformed cross-references common in government PDFs and returned no fields. PDF.js (Mozilla's engine, used in Firefox/Chrome) is more tolerant and reads field annotations successfully from the same files.

**`gemini-2.5-flash-lite`** confirmed available for this API key. Used for all text-only calls (AcroForms enrichment, field questions) — faster and cheaper than 2.5-flash for simple classification tasks.

**Key new functions:**
- `extractAcroFields(file)` — PDF.js-based, returns fields in `{id, label, value, type, sensitive}` format or null
- `uploadFileToGemini(fileData, mimeType)` — uploads to Gemini Files API via proxy, returns fileUri
- `callGemini` — now accepts optional `model` param (defaults to `GEMINI_MODEL`)

## Current scan path decision tree

```
Upload →
  Is it a PDF?
    → Try AcroForm extraction (PDF.js, no API call)
      → Fields found?
          YES → text-only call to gemini-2.5-flash-lite (tiny, fast)
                → populate domain/docType/summary/prefill → Scan confirm
          NO  → upload file to Gemini Files API → gemini-2.5-flash multimodal scan
                → auto-retry with countdown on 503
  Is it an image?
    → upload file to Gemini Files API → gemini-2.5-flash multimodal scan
```
