## What we worked on

### Demo structure review
Assessed a four-part demo structure against the current app:
1. **Happy path** ‚Äî core flow end-to-end
2. **Differentiation argument** ‚Äî side-by-side contrast with a generic chatbot
3. **Intentional failure** ‚Äî trigger something going wrong, show graceful recovery
4. **Lessons** ‚Äî talk-track, no code concern

Key finding: the intentional failure section had the most risk. Identified 503/timeout recovery as the safest failure to stage ‚Äî real retry logic already existed, realistic to trigger, no user intervention needed to recover.

---

### `?demoFail=N` ‚Äî staged 503 failure for demo

Added a URL param that lets the demo show the happy path first (no param), then the failure recovery (`?demoFail=2`):

- `_demoFailsRemaining` read once from `URLSearchParams` at page load
- `_isDemoMode` boolean captured before any decrements
- Inside the scan retry loop, if `_demoFailsRemaining > 0`, decrement and return `{ text: null, overloaded: true }` instead of calling Gemini ‚Äî no real API call made
- AcroForm fast-path bypassed when demo mode is active (`&& !_demoFailsRemaining`) ‚Äî otherwise the AcroForm path returns early and the retry loop is never reached
- Normal URL: zero effect on behaviour

### Retry UX improvements (three heuristics applied)

Applied three UX heuristics to the scan retry loop, informed by: Latency Tolerance, Transparency, and Memory Awareness.

**Latency Tolerance**
- Attempt counter shown during the live call via `thinkingEl.title`
- Countdown shows attempt number in non-demo path

**Transparency**
- Message changed from "AI service is busy" to "The AI service is temporarily overloaded ‚Äî this usually resolves in under a minute"

**Memory Awareness**
- "Your document is still loaded" added to countdown message and final failure fallback

### Demo mode retry enhancements

A suite of improvements to make the demo failure sequence feel active rather than passive. All scoped to `_isDemoMode` ‚Äî production path unchanged.

**Rotating messages with typewriter effect**
- Two message sets, one per attempt tier:
  - Attempt 1 ‚Äî optimistic: "usually clears quickly", "hang tight"
  - Attempt 2+ ‚Äî heavier: "taking longer than expected", "we haven't given up"
- Messages cycle every 5 seconds via `Math.floor((wait - t) / 5)`
- `typewrite()` fires without `await` so countdown keeps ticking independently
- 600ms `setTimeout` delay before typewrite starts ‚Äî gives previous message a moment to settle
- Typewrite speed: 25ms/char (deliberately slower than the default 15ms)

**Pip track**
- Four circles (`‚óã ‚óã ‚óã ‚óã`) rendered below the message inside `agent-brief`
- Filled circles (`‚óè`) use `#78a9ff` (IBM Blue 40 ‚Äî readable on the dark inverse background)
- Empty circles use `cds-text-disabled`
- Countdown timer rendered in the same row in `#78a9ff`, separate from the pip characters
- `minHeight: 2.5rem` pre-set on `briefText` before the element appears ‚Äî prevents height jump when typewrite clears `textContent`

**File card**
- Injected once after `scan-brief` on the first retry, lives for the full retry sequence
- Shows file icon, filename, formatted size, file type badge
- Styled with `cds-layer-01` background, `cds-border-subtle-01` border
- `cursor: pointer` + hover state (`layer-hover-01` background, stronger border)
- "View ‚Üó" affordance in `cds-link-primary` on the right edge
- Click handler: `URL.createObjectURL(state.file)` opens the file in a new tab; blob URL revoked after 10 seconds
- Removed cleanly on scan resolve via `document.getElementById('scan-file-card')?.remove()`
