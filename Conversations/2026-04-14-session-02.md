# CivicGuide — Session 02
**Date:** 2026-04-14

---

## Summary

Rebuilt the Entry stage test harness from a chat UI to a form-based wizard, per `civicguide-ux-rationale.md`. Set `test-entry.html` as the landing page. Moved the API key out of client code via a local proxy server and a Netlify serverless function. Resolved a series of model name and deployment issues to get the live site working.

---

## What Changed

### `test-entry.html` — full rewrite
Previous version: chat bubble interface with input bar and send button.

New version: form-based wizard as specified in `civicguide-ux-rationale.md`:
- **Location step** — text input + `Detect` button (browser geolocation via Nominatim reverse-geocode) + `Confirm location` CTA
- **Domain button grid** — 7 domain cards, hidden until location is confirmed
- **`#agentHint` zone** — left-border strip where Gemini's single contextual sentence surfaces; shows pulse dots while thinking
- Agent messages are silent in the UI — no chat bubbles; only `role === 'model'` responses render
- Structured messages sent to agent: `"User location: ${loc}. Respond with one calm sentence…"` / `"User selected domain: ${domain}…"`
- State panel and header retained; input bar and all chat scaffolding removed

### `index.html`
Added `window.location.replace('test-entry.html')` so the root URL forwards to the wizard. Uses `replace()` to avoid creating a history entry.

### `config.js`
- Removed `GEMINI_API_KEY` export (key moved to `.env`)
- Added `GEMINI_PROXY_URL = '/.netlify/functions/gemini'`
- Model updated to `gemini-2.5-flash` (see model troubleshooting below)

### `entry.js`
- Import changed from `GEMINI_API_KEY` to `GEMINI_PROXY_URL`
- `API_URL` changed from direct Gemini endpoint to `${GEMINI_PROXY_URL}?model=${encodeURIComponent(GEMINI_MODEL)}`

### `server.js` — new
Lightweight Node.js dev server (no npm dependencies — built-in modules only):
- Serves static files from the project root
- Proxies `POST /.netlify/functions/gemini` to the Gemini REST API, injecting the key server-side
- Reads `GEMINI_API_KEY` from `.env` (parsed manually; no dotenv package needed)
- Includes path traversal guard on static file serving
- Run with: `node server.js`

### `netlify/functions/gemini.js` — new
Netlify serverless function that proxies Gemini calls for the deployed site:
- Reads `GEMINI_API_KEY` from Netlify environment variables
- Reads model name from `?model=` query param
- Returns Gemini response body and status code verbatim

### `netlify.toml` — new
```toml
[functions]
  directory = "netlify/functions"
```

### `.env` — new
```
GEMINI_API_KEY=<key>
```
Gitignored. For local dev only.

### `.gitignore` — new
```
.env
node_modules/
```

## Deployment Troubleshooting

| Attempt | Result |
|---|---|
| `server.js` proxy via `/api/gemini` | Works locally only — no server on Netlify |
| `netlify.toml` redirect `/api/gemini` → function | 404 — redirect did not route correctly |
| Direct client call to `/.netlify/functions/gemini` | ✓ Function reached |
| Function reached, key missing | 500 — `GEMINI_API_KEY` not set in Netlify |
| Key set via `netlify env:set` + `netlify deploy --prod` | ✓ Working |

**Note:** `netlify link` was required before `netlify env:set` would work — project folder was not linked to the Netlify site.

## Decisions & Notes

- **No redirect in netlify.toml** — direct function URL in client is simpler and more reliable than a redirect that may not preserve query params
- **Model as query param** — cleaner than embedding it in the URL path; both `server.js` and the Netlify function read `?model=`
- **`server.js` matches Netlify function path** — both handle `/.netlify/functions/gemini` so `config.js` needs no environment detection
- **Geolocation** — uses browser `navigator.geolocation` + Nominatim reverse-geocode (no API key required)
